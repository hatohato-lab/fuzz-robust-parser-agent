# fuzz-robust-parser-agent 設計メモ

この文書は「なぜこの設計にしたか」の記録。使い方は README、概念入門は 01_airepo の `00_入門_…`／`00_詳説_…`。

## 実証する設計パターン
**ファジング（暗黙オラクル / implicit oracle）**。期待する「正しい出力」を一切用意せず、
**プログラムが当たり前に守るべき不変条件**（壊れない・型が崩れない）だけを定め、
**大量のランダム入力**でそれが破れないかを試す。

## なぜこの方式を選んだか（他方式との比較）★この repo の肝
- 「任意の文字列を堅牢に解釈する」処理は、入力が無限にあり**1つ1つの正解を書き出せない**。だから golden も差分テストも現実的でない。
- だが「**例外で落ちない・必ず list を返す・要素は int**」という最低限の約束は、出力を知らなくても機械的に確認できる。これが暗黙オラクル。
- そこへ**ランダム入力を大量に流す（ファジング）**ことで、人が思いつかない際どい入力で約束が破れないかを突く。
- 対比：golden/差分＝正解が1つに決まる時。メタモルフィック＝入力間の関係が言える時。モンテカルロ＝統計的性質を見る時。本 repo は「**正解は決められないが、壊れてはいけない**」列を担当。

## 確認する不変条件
- 無例外: どんな入力でも例外を外へ漏らさない。
- 返り値は list。
- list の要素はすべて int。

## なぜ信頼できるか（オラクル自己検証）
`--selftest` で、堅牢な reference は PASS、既知バグ（非数で例外を投げる／tuple を返す／文字列のまま返す）はそれぞれ別の不変条件違反として FAIL になることを示す。3 つの対照が 3 つの約束に 1 対 1 で対応する。

## 既知の限界
- ファジングは「バグが無いこと」は証明しない。**有限回で見つからなかった**だけ。N と入力分布しだい。
- 不変条件に書いていない性質（例: 解釈の正しさそのもの）は見ていない。そこは差分テストや golden の担当。
